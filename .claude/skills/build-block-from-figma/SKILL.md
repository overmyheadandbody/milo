# Build Block Skill

## UI interactions

When not in auto mode, use `AskUserQuestion` for all decisions and
confirmations — never as plain chat text.
- Binary confirmations (proceed, push branch, clean up): **Yes** / **No** options.
- Component name confirmations: **Confirm** / **Use a different name** options.
- Branch choices: present named options, **Other** for a custom name.

Initial text inputs (preview URL, Figma URLs, base branch) are
collected as regular chat prompts since they require free-form text.

In auto mode, skip confirmations per the Modes section below. The
foundation gate (Phase 1) always stops regardless of mode.

## Modes

**Auto and quiet are always ON by default.** Do not check environment
variables or run any detection steps.

The only way to change this:
- Arguments include `Auto mode: false` or `Quiet mode: false`.
- User passed `--interactive` (disables auto) or `--verbose`
  (disables quiet).

If none of the above are present, auto + quiet are on. No exceptions.

### Auto mode

Skip these confirmations and proceed automatically:

- **Phase 1 (component name)** — accept the inferred name (or the
  override from arguments) without asking.
- **Phase 1 (feature branch)** — accept the default branch name
  (`<block-name>-autogenerated`) without asking.
- **Phase 8 (cleanup)** — auto-clean `.playwright-mcp` and
  `/tmp/build-block-figma/` without asking.

**Never skip in auto mode:**
- **Phase 1 (foundation gate)** — if `foundation: c2` is missing, always
  stop and tell the user.
- **Auth / push failures** — always stop on git or CDN errors.

### Quiet mode

- Emit short, jargon-free progress updates ("Reading Figma designs...",
  "Writing component code...", "Comparing against designs...",
  "Checking accessibility...", "Running speed test...").
- Do not output CSS variable lists, token names, code snippets, git
  commit hashes, axe-core rule IDs, or Lighthouse metric tables.
- Accessibility and performance: "Passed" or "Issues found: [description]".
- Final summary: block name, test URL, feature branch name, pass/fail
  for accessibility and performance, and next steps only.

---

You are building a new block component for the **C2 design system** inside
the `adobecom/milo` repository.  Components are plain HTML blocks styled
with plain CSS and initialised by an external orchestrator that calls
`init()` on load — you never add self-initialisation logic.

> **Critical path rules**
>
> - Block code patterns come **exclusively** from `libs/c2/blocks/`.
>   Never reference or imitate blocks under `libs/blocks/`.
>   **Never copy or create files in `libs/blocks/`** — if the page
>   does not load C2 blocks, the foundation metadata is missing
>   (see Phase 1 gate).  Do not work around this.
> - Shared utilities live in `libs/utils/decorate.js`
>   (`decorateBlockText`, `decorateTextOverrides`, and the viewport
>   content parser) and `libs/utils/utils.js` (`createTag`).
>   Re-use these; do not duplicate their logic.
> - Design tokens live in `libs/c2/styles/styles.css`.
> - The platform is **Adobe EDS**.  Use the **Fluffyjaws MCP** to look
>   up any EDS conventions you are unsure about.

## Phase 0 — Preview URL resolution

After collecting the preview URL, determine its type and resolve it.

### Localhost (`http://localhost:...`)

No special handling. Proceed directly to Phase 1.

### DA preview (`.aem.page`)

A `.aem.page` URL will not work with `?milolibs=`. The user must
provide a published (`.aem.live`) URL or let Claude publish via the
EDS admin API. If publishing, **load
`references/remote-branch-workflow.md` section 1 now**.

After obtaining the `.aem.live` URL, fall through to DA published
below.

### DA published (`.aem.live`)

Parse org, repo, and page path from the URL:
```
https://main--<repo>--<org>.aem.live/<path>
```

Store these values. Set **`remote-branch-mode = true`**.

> **STOP**: Do NOT proceed to Phase 1 until you have either a
> localhost URL or a resolved `.aem.live` URL. The resolved URL is
> referred to as the **page URL** from here on.

## Phase 1 — Validate environment & infer component name

Use Playwright MCP to navigate to the page URL.

### Foundation check (gate)

Verify the page has `<meta name="foundation" content="c2">` in
`<head>`.

If **missing**, stop and ask the user to add `foundation: c2`
metadata. Do not work around this by copying files to `libs/blocks/`.

**Remote-branch-mode recovery**: the user must add the metadata in
DA, re-preview, and re-publish. Offer to handle preview/publish via
the EDS admin API (`remote-branch-workflow.md` section 1).

### Infer component name

Inspect the DOM inside `<main>` only. Look for the block's
distinctive class name (e.g. `class="hero-banner"`). Derive the
component name in **kebab-case**.

**Auto mode**: use the inferred name and proceed without asking.

**Default**: confirm the inferred name with the user before continuing.

### Create feature branch (remote-branch-mode only)

Skip if `remote-branch-mode` is `false`.

**Load `references/remote-branch-workflow.md` section 2 now** and
follow the branch creation procedure.

**Auto mode**: create the branch and proceed without asking.

**Default**: do NOT proceed to Phase 2 until the user confirms
the branch.

## Phase 2 — Read existing patterns

Before writing any code:

1. **List `libs/c2/blocks/`** and read **2–3 blocks**. Study folder
   structure, `init(el)` patterns, CSS namespacing, and DOM interaction.
2. **Load `references/eds-patterns.md` now.**
3. Read `libs/c2/styles/styles.css` — extract all CSS custom properties,
   especially `--s2a-` tokens.
   **Load `references/design-tokens.md` now.**
4. Read `libs/utils/decorate.js` for `decorateBlockText`,
   `decorateTextOverrides`, and the viewport content parser.
   **Load `references/viewport-content.md` now** — but only if the
   block's authored DOM contains viewport keywords. Otherwise skip.
5. Read `libs/utils/utils.js` for `createTag` and other helpers.
6. Skim `libs/c2/blocks/section-metadata/section-metadata.css` for
   the grid/container system.
   **Load `references/grid-system.md` now.**

## Phase 3 — Read Figma designs

Use the **Figma MCP** to retrieve each provided frame URL.
For each frame, extract:

- Layout structure (flex / grid, direction, gaps, padding, margins)
- Typography (font family, size, weight, line height, letter spacing)
- Colours — match against `--s2a-` CSS variables
- Spacing — match against spacing tokens
- Imagery, iconography, and decorative elements
- Stacking and element ordering — Figma layers top-to-bottom often
  mean vertical stacking, not horizontal. Check auto-layout direction.

If multiple frames are provided, explicitly note layout differences
between breakpoints.

### Cache Figma frames to disk

Save each frame image to `/tmp/build-block-figma/` (`mobile.png`,
`tablet.png`, etc.) so it remains available during Phase 5 if the
Figma MCP session expires.

### Build a per-breakpoint element inventory

Compile a comparison table listing every visible element at each
breakpoint:

```
Element         | Mobile | Tablet | Desktop
────────────────────────────────────────────
Heading         | ✓      | ✓      | ✓
CTA button      | ✓      | ✓      | ✓
Hero image      | ✓      | ✓      | ✓
Background      | solid  | gradient| gradient
```

If an element appears in any Figma frame, flag which breakpoints
include it. Carry this inventory into Phase 4 as a checklist.

## Phase 4 — Build the component

### Check out the working branch (remote-branch-mode only)

Skip if `remote-branch-mode` is `false`.

**Load `references/remote-branch-workflow.md` section 3 now** and
follow steps 1-2 only (fetch and check out temporary local branch).
Do **not** commit or push yet.

### Create block files

Create:
- `libs/c2/blocks/<name>/<name>.js`
- `libs/c2/blocks/<name>/<name>.css`

### Register the block in the C2 block list

Add the new block's kebab-case name to the `C2_BLOCKS` array in
`libs/utils/utils.js`. **Mandatory** — without it the EDS loader
falls back to `libs/blocks/`.

### JS and CSS rules + acceptance criteria

**Load `references/acceptance-criteria.md` now.** Follow every item.

### Lint and fix

```bash
npx eslint libs/c2/blocks/<name>/<name>.js --fix
```

Resolve any remaining errors manually before proceeding.

### Commit and push (remote-branch-mode only)

Skip if `remote-branch-mode` is `false`.

Complete `remote-branch-workflow.md` section 3 (steps 3-4): commit
JS, CSS, and `utils.js`, push, clean up temporary local branch.

## Phase 5 — Visual comparison loop

### 5a. Refresh DA preview (localhost only)

Skip if `remote-branch-mode` is `true`.

If you modified DA HTML (e.g. adding metadata in Phase 1):
```bash
curl -s -X POST \
  "https://admin.hlx.page/preview/<org>/<repo>/main/<page-path>"
```

### 5b. Construct the Playwright URL

**Localhost**: use the page URL as-is.

**Remote-branch-mode**: `<page-url>?milolibs=<branch-name>`

This is the **Playwright URL** for all subsequent steps.

### 5c. Force-refresh after code push (remote-branch-mode only)

Skip if `remote-branch-mode` is `false`.

Follow `references/remote-branch-workflow.md` section 4.

### 5d. Pre-flight check

Navigate to the Playwright URL and verify the block loaded:

1. `document.querySelector('.<block-name>')` exists.
2. `data-block-status` equals `"loaded"`.
3. Expected decorated classes are present.
4. No block-loading errors (403/404) in console.

**Common failures:**
- **Block JS 404/403** from site path instead of
  `/libs/c2/blocks/...`: `foundation: c2` meta missing or block
  not in `C2_BLOCKS`.
- **Fails on reload**: stale cached `utils.js`. Ensure Playwright
  cache bypass is active (`remote-branch-workflow.md` section 4b).
- **`data-block-status` null**: block JS never fetched. Check
  network/console.
- **(Remote) 404 on feature branch files**: CDN not indexed yet.
  Trigger per-file code preview (section 4a), wait 10-15s, reload.

Do **not** proceed to screenshots until pre-flight passes.

### 5e. Visual comparison

**Load `agents/visual-comparison.md` now.**

For each breakpoint, screenshot and compare against the cached Figma
frame. Maximum **5 passes** total. Stop early if fidelity is high.

One pass = identify issues → fix → (remote: batch + push + refresh)
→ re-screenshot. See `remote-branch-workflow.md` section 5 for
batching criteria.

Only after the visual loop is complete, proceed to Phase 6 and 7.

---

## Phase 6 — Accessibility audit

**Load `agents/accessibility-check.md` now** and follow its procedure.

Run axe-core against the block's container element. Fix WCAG 2.2 AA
violations. If fixes require code changes and `remote-branch-mode`
is `true`, push via `remote-branch-workflow.md` section 3, then
force-refresh per section 4.

---

## Phase 7 — Performance audit

**Load `agents/performance-check.md` now** and follow its procedure.

Run Lighthouse against the Playwright URL. Assess LCP impact. If
fixes require code changes and `remote-branch-mode` is `true`, push
and force-refresh as in Phase 6.

---

## Phase 8 — Summary

Output:

1. **Component name** and file paths created.
2. **Feature branch** (remote-branch-mode only) — name, repo, commits.
3. **Breakpoints implemented** and corresponding Figma frames.
4. **CSS variables used** — every `--s2a-` token referenced.
5. **Hardcoded values** — Figma values with no matching token.
6. **Acceptance criteria checklist** — confirm each passes.
7. **Accessibility results** — axe-core summary.
8. **Performance results** — Lighthouse score, LCP, CLS, INP, TBT.
9. **Obstacles Encountered** — aggregated from all subagents.

### Next steps (remote-branch-mode only)

- **Open a PR** from `<branch-name>` into `stage`.
- **Test in context** at `<page-url>?milolibs=<branch-name>`.
- **Delete the feature branch** if exploratory.

```bash
rm -rf .playwright-mcp /tmp/build-block-figma
```

**Default**: ask the user whether to remove `.playwright-mcp` and
`/tmp/build-block-figma/`. Delete on confirmation, keep otherwise.
